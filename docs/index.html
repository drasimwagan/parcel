Describe the thing you need. Get a working module.

Parcel is an AI-native, modular business-application platform. An admin writes a business need in plain language. Parcel drafts a module — models, views, migrations, tests — runs it in a sandbox, shows a preview, and installs it on approval. Developers can hand-write the same kind of module with a typed Python SDK when precision matters. End users never see the authoring layer.

What Parcel is

Think Odoo, redesigned around large-language-model authoring. The shell is a small, well-typed FastAPI service that does auth, RBAC, the admin UI, and the module lifecycle. Everything else is a module — a Python package, pip-installable, discovered via entry points, owning its own Postgres schema and Alembic migrations.

Shell

FastAPI, HTMX, Postgres, Redis. Auth, RBAC, admin UI, module loader, AI authoring pipeline. Knows zero module names.

SDK

The only thing modules import. Stable, versioned surface: models, permissions, views, migrations.

Modules

Pip-installable packages. Own their schema (mod_<name>) and migrations. Capabilities declared in the manifest; admins approve at install time.

How the AI generator will work

Landing in Phase 7. The flow is deliberately uninteresting — because "interesting" in this space is how you lose people's data.

  1. Admin chats with Claude about what they want.
  2. Claude drafts a module against the SDK.
  3. Draft runs through ruff + bandit + an AST policy that blocks os, subprocess, socket, eval, exec, and dynamic imports unless the manifest declares the capability.
  4. If the static gate passes, the module is installed in a sandbox database.
  5. Admin sees a preview — sample records, screenshots of views, the migration plan.
  6. On approval, the module installs into production. Capabilities are re-confirmed.

Quickstart

Requires Docker. Clone, bring up dependencies, migrate, run.

git clone https://github.com/drasimwagan/parcel.git
cd parcel
cp .env.example .env                       # edit PARCEL_SESSION_SECRET for non-dev use
docker compose up -d postgres redis
docker compose run --rm shell migrate      # creates the `shell` schema
docker compose up -d shell

Smoke-check:

curl http://localhost:8000/health/live     # {"status":"ok"}
curl http://localhost:8000/health/ready    # {"status":"ok","checks":{"db":"ok","redis":"ok"}}

Scaffold and install a module with the parcel CLI (Phase 6):

uv sync --all-packages
uv run parcel new-module widget            # writes modules/widget/
uv run parcel install ./modules/widget     # pip install -e + activate + migrate
uv run parcel dev                          # uvicorn with --reload on :8000

Sandbox a candidate module through the static-analysis gate (Phase 7a):

uv run parcel sandbox install ./modules/contacts  # gate + install → /mod-sandbox/<uuid>/
uv run parcel sandbox list                         # see status + gate verdict
uv run parcel sandbox promote <uuid> new_mod       # copy files → real install

Generate a module from a natural-language prompt (Phase 7b — needs ANTHROPIC_API_KEY):

uv run parcel ai generate "track invoices with number, amount, date"
# ✓ sandbox <uuid> at /mod-sandbox/<short>/

Tests use testcontainers to spin up an ephemeral Postgres, so Docker must be running:

uv run pytest                              # 259 tests, ~60s

Design principles

Small, typed surface

Python 3.12, pyright strict on the shell and SDK. Every public function has a test. No raw SQL in application code.

Isolation by default

Each module owns its schema. Shell never imports from a module. Modules only import parcel-sdk — the shell registers its implementation with shell_api.bind() at startup, so the dep direction never inverts.

AI is sandboxed

Generated code never touches the production database without an admin preview. Capabilities must be declared and approved.
