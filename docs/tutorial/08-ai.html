08 Authoring modules with AI

You've just written a Notes module by hand. Now try the other path — describe what you want at /ai and have Claude draft it for you. The generator is bound by the same rules you just followed: modules import only parcel_sdk, the static gate enforces it, and anything that slips past the AI lands in a sandbox for your inspection.

Prerequisites

Your shell needs an AI provider configured. The default is PARCEL_AI_PROVIDER=api, which requires ANTHROPIC_API_KEY in your environment (or .env). Alternatively, set PARCEL_AI_PROVIDER=cli and ensure the Claude Code CLI is on your PATH.

Without either, the generator endpoint returns 503 and the chat UI shows a "provider not configured" banner. The shell still boots — the AI piece is optional infrastructure.

Open the chat surface

  1. Log into the admin UI.
  2. Click "Generator" in the "AI Lab" section of the sidebar (or visit /ai).
  3. Click "+ New session".
  4. Type a prompt. Hit Send.

The page shows a spinner while a background task runs the generator. HTMX polls every second; when the task finishes the page updates automatically. Typical generation takes 30-90 seconds.

Anatomy of a turn

A single "turn" is one prompt → one module attempt:

  1. Your prompt is sent to the configured ClaudeProvider (API or CLI) with the system prompt that ships in the repo at packages/parcel-shell/src/parcel_shell/ai/prompts/generate_module.md.
  2. Claude emits a series of write_file(path, content) tool calls followed by one submit_module(). The API provider validates every path (no absolute paths, no .., no .sh/.exe, 64 KiB per file, 1 MiB total).
  3. The shell zips the files and hands them to create_sandbox — the same Phase 7a pipeline you used manually.
  4. If the gate passes, you get a sandbox row with a green card and a link to try it live.
  5. If the gate rejects, the shell retries once with the gate report attached as context ("your last attempt failed — here's what went wrong"). If the second attempt still fails, the turn is marked exceeded_retries and the gate report is shown on the card.

Prompts that pass first time

The system prompt is opinionated — it tells Claude exactly what files to emit, which imports are allowed, which calls are hard-blocked, and how to declare capabilities. Your job is to describe the business logic: what the module tracks, what the user can do with it, what permissions/views make sense. Don't tell Claude which Python packages to use — the system prompt handles that.

Good prompt:

Track invoices. An invoice has a number (string), an amount (decimal), an issued date, and a paid status. Users can list invoices, create them, and mark them paid. Two permissions: invoices.read and invoices.write.

Bad prompt (too much Python specifics, Claude will get confused):

Use SQLAlchemy to make a table with httpx and pandas for data analysis...

Also bad (too vague):

Make an invoice module.

The sweet spot: describe the entity, its fields, the user-visible operations, and the permission shape. Claude fills in the rest.

When the gate rejects

Click the turn's "Gate report" disclosure to see exactly which rule flagged which file and line. The most common rejections:

  • ast.blocked_import.os — Claude reached for os.path.join. Use pathlib.Path instead, or declare capabilities=("filesystem",) if you genuinely need filesystem access.
  • ast.forbidden_package.parcel_shell — Claude imported from parcel_shell.*. Re-prompt reminding it to use parcel_sdk.shell_api.
  • ast.raw_sql — Claude used sqlalchemy.text(...). Either rewrite with the ORM or declare capabilities=("raw_sql",) and re-prompt.
  • ast.unknown_package (warning) — an imported top-level package isn't on the allow-list. Usually safe to ignore; if it's a real dep you want, add it manually after the sandbox lands and re-run the gate via parcel sandbox install.

Iterating within a session

Each turn is an independent generation. Claude doesn't see prior turns in this session when drafting the next one — the "conversation" is a UX pattern, not a model-context pattern. So if your second prompt is "same as before but add a due_date field", Claude will interpret that as the entire spec and might drop fields from the first attempt.

Two alternative patterns:

  • Promote the good turn. Inspect the sandbox, promote it to a real install, then edit the promoted module by hand to add the extra field. You get AI-drafted scaffolding + human-controlled iteration.
  • Re-prompt fully. "Track invoices. An invoice has a number, amount, issued date, due date, and paid status..." — repeat the entire spec. Each turn becomes its own standalone generation.

When to use the generator

  • Greenfield modules with straightforward CRUD. Claude is good at producing the scaffold + a working first pass for simple entities.
  • Exploratory prototypes. "What would a tasks module look like?" can be answered in two minutes instead of twenty.
  • Onboarding new entity fields. Not yet supported programmatically, but a workable pattern today: promote the draft, write tests for it, then hand-edit.

The generator is not a replacement for hand-written modules when you care about precise UX, complex business rules, or specific performance characteristics. It's a scaffolding tool that produces something that passes the gate; the human-in-the-loop review at the sandbox stage is load-bearing, not a rubber stamp.

Promoting an AI draft

Exactly the same as Path 3 in the previous step. Inspect at /sandbox/<uuid>, try it live at /mod-sandbox/<short-uuid>/, then either: 

# via CLI
uv run parcel sandbox promote <uuid> invoices

# or via the admin UI
# /sandbox/<uuid> → fill the promote form → Submit

Either way the capability-approval check runs: you're re-confirming what Claude declared, and the install fails cleanly if the two don't match.
