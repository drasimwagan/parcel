Build your first module

The Parcel module tutorial

Every Parcel module — handwritten or AI-generated — is a small, pip-installable Python package that plugs into the shell through a stable SDK. This tutorial walks you through the whole surface, using a Notes module as the running example. By the end you will have written a working module that lives at /mod/notes/, passes the static-analysis gate, and installs into a sandbox for review before it ever touches production data.

What you'll build

A simple Notes module. One entity (note: title, body, timestamp), one sidebar entry, two permissions (notes.read, notes.write), list + create + detail + delete views, and a smoke test. Small enough to read in a sitting, big enough to touch every concept you'll use in real modules.

The existing Contacts module under modules/contacts/ is a larger reference you can grep through after you've worked through the tutorial. The patterns are the same.

Prerequisites

  • A working Parcel checkout with uv sync --all-packages succeeding.
  • Docker running, with Postgres reachable (the quickstart in the main README covers this).
  • A seeded admin user (docker compose run --rm shell bootstrap create-admin …).
  • Comfort with Python 3.12, FastAPI, and SQLAlchemy 2.0 async. You don't need prior SQLAlchemy experience — the model code we write is straightforward — but this tutorial doesn't teach the basics.

Contents

  • 01 Foundations What a Parcel module actually is. Architecture, entry points, the SDK facade, the shell–module boundary.
  • 02 Scaffolding with the CLI Use parcel new-module notes to generate the skeleton. Walk through every file it creates and why it exists.
  • 03 Data layer — models and migrations Define the Note entity, bind it to the mod_notes schema, and write the initial Alembic migration.
  • 04 HTTP layer — router, shell_api, templates Write list / create / detail / delete routes using parcel_sdk.shell_api. No direct shell imports — ever.
  • 05 Permissions, sidebar, capabilities Declare notes.read / notes.write, add a sidebar entry, decide whether to declare any capabilities.
  • 06 Testing Write a smoke test, a service test, and an HTTP integration test. Tests are allowed to import parcel_shell.* directly.
  • 07 Installing — the four paths Direct install via the CLI, install via the admin UI, sandbox install (gated), promote from sandbox to real.
  • 08 Authoring modules with AI Use /ai to draft a module from a prompt. What the gate will reject, how to word a prompt so it passes first time.
