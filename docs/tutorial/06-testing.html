06 Testing

Parcel tests run against a real Postgres (via testcontainers), so your service and router tests exercise the same migrations, the same schema, and the same RBAC that production does. Tests are also the one place where you're allowed to import parcel_shell.* directly — the SDK-only constraint is a runtime rule, not a test rule.

Three test layers

  1. Smoke — imports your module, asserts manifest identity. Already in place from the scaffold.
  2. Service — exercises service.py directly against a real DB session. Fast, focused.
  3. Router — HTTP integration, goes through auth + permissions + templates. Slower, end-to-end.

Start with service tests; they're where most of your real logic lives. Add router tests for the happy-path of each endpoint and for known edge cases.

Add parcel-shell as a dev dep

Tests need parcel_shell to boot the app, create users, and so on. Tests are exempt from the SDK-only rule, but you still shouldn't pull parcel-shell into your runtime dependency list. Add it to a dev group in pyproject.toml: 

[dependency-groups]
dev = ["parcel-shell"]

[tool.uv.sources]
parcel-sdk = { workspace = true }
parcel-shell = { workspace = true }

Now uv sync --all-packages makes parcel_shell importable during tests, but a wheel build of your module doesn't pull it in as a runtime requirement.

Service tests

Create tests/test_notes_service.py: 

from __future__ import annotations

import pytest
from sqlalchemy.ext.asyncio import AsyncSession

from parcel_mod_notes import service
from parcel_mod_notes.models import Note


@pytest.mark.asyncio
async def test_create_note_persists(db_session: AsyncSession) -> None:
    note = await service.create_note(
        db_session, title="Ideas", body="Some body text"
    )
    assert note.id is not None
    assert note.title == "Ideas"

    reloaded = await db_session.get(Note, note.id)
    assert reloaded is not None
    assert reloaded.title == "Ideas"


@pytest.mark.asyncio
async def test_list_notes_ordered_by_created_at(db_session: AsyncSession) -> None:
    await service.create_note(db_session, title="first", body="")
    await service.create_note(db_session, title="second", body="")
    notes = await service.list_notes(db_session)
    # Newest first.
    assert [n.title for n in notes[:2]] == ["second", "first"]


@pytest.mark.asyncio
async def test_delete_note_removes_row(db_session: AsyncSession) -> None:
    note = await service.create_note(db_session, title="gone", body="")
    await service.delete_note(db_session, note=note)
    reloaded = await db_session.get(Note, note.id)
    assert reloaded is None

db_session is a fixture provided by the workspace's shared test plugin (_shell_fixtures.py). It gives you a real async session wrapped in a savepoint that rolls back at teardown, so each test runs against a clean schema without paying a full migration cost per test.

Router tests

Router tests use the committing_admin fixture — an httpx.AsyncClient already logged in as a freshly-created admin. The shell's lifespan is up, your module is mounted, the auth middleware is real.

Create tests/test_notes_router.py: 

from __future__ import annotations

import pytest
from httpx import AsyncClient


@pytest.mark.asyncio
async def test_list_page_renders(committing_admin: AsyncClient) -> None:
    r = await committing_admin.get("/mod/notes/")
    assert r.status_code == 200
    assert "Notes" in r.text
    assert "New note" in r.text


@pytest.mark.asyncio
async def test_create_then_detail(committing_admin: AsyncClient) -> None:
    r = await committing_admin.post(
        "/mod/notes/",
        data={"title": "Ideas", "body": "hello"},
        follow_redirects=False,
    )
    assert r.status_code == 303
    detail_url = r.headers["location"]
    assert detail_url.startswith("/mod/notes/")

    r = await committing_admin.get(detail_url)
    assert r.status_code == 200
    assert "Ideas" in r.text
    assert "hello" in r.text


@pytest.mark.asyncio
async def test_create_rejects_empty_title(committing_admin: AsyncClient) -> None:
    r = await committing_admin.post(
        "/mod/notes/", data={"title": "", "body": ""}, follow_redirects=False
    )
    assert r.status_code == 303
    assert r.headers["location"].endswith("/new")


@pytest.mark.asyncio
async def test_nonexistent_note_is_404(committing_admin: AsyncClient) -> None:
    r = await committing_admin.get(
        "/mod/notes/00000000-0000-0000-0000-000000000000"
    )
    assert r.status_code == 404

The gate will check your test code too — no, wait, it won't

A common misconception: because Claude-generated modules go through the static gate, and the gate scans .py files, surely tests are scanned too? They're not. parcel_gate.runner explicitly skips tests/, __pycache__/, .venv/, and similar directories. Your tests can import httpx, from parcel_shell.something import thing, use sqlalchemy.text(...) — none of that matters for gate verdicts. The gate is a runtime-code safety check.

Run them

uv run pytest modules/notes/ -v

On first run, testcontainers pulls a Postgres image (~100 MB one-time) and spins up a fresh container. Subsequent runs reuse it. Expect ~5-10s startup for the first test, then fast.

Lint + type-check

uv run ruff format
uv run ruff check
uv run pyright

All three should pass. Modules are included in the root pyright config at basic-check level; the shell and SDK run at strict. If you introduce a new Python dep, add it to pyproject.toml and uv sync --all-packages again.

When tests fail but the sandbox passes (or vice versa): they're checking different things. Tests exercise your logic against a real DB. The sandbox install exercises the gate, your migrations, your router mounting. Run both on every meaningful change.
