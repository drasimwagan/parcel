05 Permissions, sidebar, capabilities

Our router already references notes.read and notes.write, but they don't exist yet. Permissions are part of the module's manifest — declare them there and the shell registers them into the RBAC system and attaches them to the admin role at install time. The sidebar works the same way.

Declare the permissions

Open src/parcel_mod_notes/__init__.py and add the two permissions. While we're here we'll also add a sidebar item. 

from __future__ import annotations

from pathlib import Path

from parcel_mod_notes.models import metadata
from parcel_mod_notes.router import router
from parcel_sdk import Module, Permission, SidebarItem

module = Module(
    name="notes",
    version="0.1.0",
    permissions=(
        Permission("notes.read", "View notes"),
        Permission("notes.write", "Create, update, and delete notes"),
    ),
    capabilities=(),
    alembic_ini=Path(__file__).parent / "alembic.ini",
    metadata=metadata,
    router=router,
    templates_dir=Path(__file__).parent / "templates",
    sidebar_items=(
        SidebarItem(label="Notes", href="/mod/notes/", permission="notes.read"),
    ),
)

__all__ = ["module"]

What happens at install time

When an admin installs your module (via the CLI, the admin UI, or a sandbox promote), the shell's install_module service:

  1. Creates the mod_notes schema.
  2. Upserts your declared permissions into shell.permissions.
  3. Attaches every declared permission to the built-in admin role (so admins can immediately use the module).
  4. Runs your Alembic migrations against the new schema.
  5. Records a row in shell.installed_modules with your version, approved capabilities, and current head revision.
  6. Mounts your router and prepends your templates dir on the live app.

You don't write any of this logic. The shell does it because you declared a manifest.

Assigning permissions to other roles

Admin gets every permission by default. For any other role — say, "staff" — an admin assigns notes.read (and optionally notes.write) through the admin UI at /roles. Your module doesn't need to care: the shell's RBAC system handles the storage, the require_permission dep checks at request time, and the composed sidebar filters out items the user can't see.

The sidebar item

We declared one: 

SidebarItem(label="Notes", href="/mod/notes/", permission="notes.read")

Three fields. The permission is optional — if omitted, the item is shown to every logged-in user. Ours is gated on notes.read, which means users who lack that permission won't see "Notes" in the sidebar. (They also can't reach /mod/notes/ because of the require_permission dep — the sidebar filter is UX, not security.)

Modules get their own section in the sidebar, labelled with the module's name. So after install, the sidebar grows a "Notes" section with one "Notes" entry inside it. Multi-entry modules (like Contacts, which has "Contacts" and "Companies") get multiple SidebarItem tuples.

Capabilities

Our Notes module declares capabilities=() — an empty tuple. That's the correct default for any business-CRUD module. Only declare a capability when your module genuinely needs it, and even then, expect the admin to scrutinise it at install time.

The four values:

  • "filesystem" — unlocks import os and calls to open().
  • "process" — unlocks import subprocess.
  • "network" — unlocks socket, urllib, http.*, httpx, requests, aiohttp.
  • "raw_sql" — unlocks sqlalchemy.text(...).

If you need to POST to an external API, you'd declare capabilities=("network",). The gate will then allow import httpx in your module; without it, the gate rejects the install. Admins see the capability list at install time and decide whether to approve.

Some things are always blocked. Regardless of what you declare, the gate rejects import sys, import importlib, any import from parcel_shell.*, calls to eval/exec/compile/__import__, and access to sandbox-escape dunders like __subclasses__. There is no capability that unlocks these. If you genuinely need that behaviour, you're writing platform code (it goes in parcel-shell), not a module.

Verify

uv run pytest modules/notes/
uv run parcel sandbox install ./modules/notes

Sandbox install still passes. The gate now sees your two permissions and the sidebar item — declared, not used in any risky way, so no finding.

Try it in the admin UI

# In one terminal, keep the dev server running:
uv run parcel dev

# In another, install into the real schema (not sandbox):
uv run parcel install ./modules/notes
uv run parcel migrate

Refresh http://localhost:8000/. The sidebar now has a "Notes" section (scroll down in the left nav). Click "Notes", you should land on an empty list page. "+ New note" takes you to the form. Submit it. You should land on the detail page with a success flash.

If any of that fails, check the shell logs — they're structured JSON in non-dev envs, readable text in dev.
