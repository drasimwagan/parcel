07 Installing — the four paths

There are four ways a Parcel module can be installed, depending on whether you trust the source and whether you want to try it before promoting it to real. Understanding when to use which is the difference between a comfortable dev loop and a risky one.

The four paths at a glance

PathGate?SchemaUse for
parcel install <path>NoReal mod_<name>Your own code. Fast dev loop.
POST /admin/modules/installNoReal mod_<name>Admin-initiated, already-discovered modules.
parcel sandbox install <path>YesSandbox mod_sandbox_<uuid>Candidates you want to review before promoting.
/ai chat + gate + sandboxYesSandbox mod_sandbox_<uuid>AI-authored modules. Promote only after inspection.

Path 1 — Direct CLI install

uv run parcel install ./modules/notes

The CLI uv pip install -e's your module into the active venv, boots the shell in-process (via asgi-lifespan's LifespanManager), discovers your module by entry point, and calls the same install_module service function that the admin HTTP endpoint uses. No gate, no sandbox — you authored the code, you trust it.

Ideal iteration loop: 

uv run parcel new-module widgets
# … edit files …
uv run parcel install ./modules/widgets       # real install
# … edit more files, bump version in pyproject.toml and module.py …
uv run parcel migrate --module widgets        # pick up new migration

Path 2 — Admin UI

Modules discovered on the shell (i.e., pip installed and therefore visible via entry points) show up at /modules in the admin UI with an "available" pill. Click through to the detail page and hit "Install". You get a form to approve each declared capability (empty for Notes), then submit. Same underlying code path as parcel install.

Path 3 — Sandbox install

uv run parcel sandbox install ./modules/notes

Same lifecycle, but routed through parcel_shell.sandbox.service.create_sandbox:

  1. Your source is extracted to var/sandbox/<uuid>/.
  2. The manifest is parsed (enough to read the declared capabilities; the module itself is not imported yet).
  3. parcel-gate runs: ruff + bandit + AST policy. Tests and caches are skipped — the gate examines runtime code only.
  4. On pass: a throwaway schema mod_sandbox_<short-uuid> is created, your migrations run against it (with metadata.schema patched in memory), your module is loaded via importlib.util under a unique sys.modules name so two sandboxes of the same module can coexist, and your router mounts at /mod-sandbox/<short-uuid>/.
  5. A row is written to shell.sandbox_installs, visible at /sandbox in the admin UI.

The sandbox has a 7-day TTL. Run parcel sandbox prune periodically to dismiss expired sandboxes.

Promoting a sandbox to real

uv run parcel sandbox promote <uuid> notes_v2 -c network

Promote does three things: (1) copies the sandbox files to modules/notes_v2/ and rewrites package-name references if the target name differs from the sandbox's, (2) uv pip install -e's the target, (3) runs the real install path against the approved capabilities. The sandbox is dismissed after a successful promote.

Capabilities approved at promote (-c network above) must exactly equal the module's declared capabilities — not a subset, not a superset. This guards against "I saw network declared but I'll pretend I didn't" bugs.

Dismissing a sandbox

uv run parcel sandbox dismiss <uuid>

Drops the sandbox schema, removes the files, marks the row dismissed (kept for audit). The router stays mounted until the next shell restart (FastAPI limitation) but every request 500s because the schema is gone — acceptable for the lifespan of a single dev session.

Path 4 — AI-authored via /ai

Covered in the next step. The short version: describe what you want in plain language; the generator drafts a module; the draft goes through the gate; if it passes, it lands as a sandbox install. From there it's Path 3 — inspect, promote, or dismiss.

Upgrades

Bump your Module(version=...), add a new Alembic revision, uv sync --all-packages, then: 

uv run parcel migrate --module notes

Migration-head is stored on the installed-module row, so re-running migrate without a new revision is a no-op. The per-module Alembic version table lives inside your schema (mod_notes.alembic_version), so you can downgrade past the shell baseline cleanly.

Uninstall

# Soft — flip is_active=false, keep data
curl -b cookies.txt -X POST http://localhost:8000/admin/modules/notes/uninstall

# Hard — downgrade + drop schema
curl -b cookies.txt -X POST 'http://localhost:8000/admin/modules/notes/uninstall?drop_data=true'

Soft uninstall is idempotent and safe — the next shell boot just skips mounting the module. ?drop_data=true is a last-resort; it runs your downgrade(), drops the schema, removes the installed-module row. There is no built-in undo.

Docker-compose production deploy

docker compose up -d postgres redis
docker compose run --rm shell migrate      # shell schema only
docker compose up -d shell

# From outside the container, while shell is up:
uv run parcel install <git-url-of-your-module>
uv run parcel migrate --module <name>

In production you'll typically install modules from Git URLs via uv pip install. The CLI's install accepts a Git URL directly when the source argument isn't a local path.
