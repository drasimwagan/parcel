01 Foundations

Before we scaffold a single file, it's worth being precise about what a Parcel module is. That framing makes the rest of the tutorial — and every design choice the platform enforces — obvious instead of arbitrary.

The one-paragraph definition

A Parcel module is a pip-installable Python package that contributes a FastAPI router, its own Postgres schema, its own Alembic migrations, a set of permissions, and optionally some Jinja templates and sidebar entries — all through a stable SDK surface called parcel_sdk. The shell discovers modules via Python entry points, installs them on admin request, runs their migrations against their dedicated schema, and mounts their routes under /mod/<name>/.

The three-layer architecture

Parcel is intentionally three layers, with strict rules about who imports what. 

┌────────────────────────────────────────────────────────────┐
│  parcel-shell  (the FastAPI app)                           │
│  auth · RBAC · admin UI · module loader · AI pipeline      │
│                                                            │
│  Never imports from any module by name.                    │
├────────────────────────────────────────────────────────────┤
│  parcel-sdk  (the stable API modules import)               │
│  Module, Permission, SidebarItem, shell_api (6 functions)  │
│                                                            │
│  The shell fills this in at startup via shell_api.bind().  │
├────────────────────────────────────────────────────────────┤
│  modules  (pip packages, entry-point discovered)           │
│  parcel_mod_contacts · parcel_mod_notes · …                │
│                                                            │
│  Only ever import parcel_sdk.*                             │
└────────────────────────────────────────────────────────────┘

The shell is what users log into. The SDK is the only surface your module talks to. Modules never reach inside the shell — not into parcel_shell.db, not into parcel_shell.ui.*, nothing. When a module needs a database session, a permission check, or the shared template loader, it goes through parcel_sdk.shell_api.

Why this matters. Modules can be AI-generated and installed sight-unseen into a sandbox. The static-analysis gate (Phase 7a) rejects anything that imports parcel_shell.* because that would let an AI-authored module reach around the facade and do whatever it wanted. The SDK boundary is load-bearing for safety, not just aesthetics.

What the SDK gives you

parcel_sdk exports roughly six useful things:

  • Module — the manifest dataclass. Name, version, permissions, router, templates dir, sidebar items, Alembic ini, metadata.
  • Permission — a named permission + description.
  • SidebarItem — a label + href + optional permission gate.
  • run_async_migrations — the helper your Alembic env.py calls.
  • shell_api — the facade: get_session(), require_permission(name), effective_permissions(request, user), set_flash(response, flash), get_templates(), sidebar_for(request, perms).
  • shell_api.Flash — frozen dataclass for one-shot UI banners (kind="success"|"error"|"info").

Every handler in every module lives inside these six functions. If you find yourself wanting something that isn't there, the answer is almost always "write a service function inside your module" — the shell's job is to provide primitives, not application logic.

The module lifecycle

  1. Author. You parcel new-module <name> (or describe it to the AI at /ai). Files land in modules/<name>/.
  2. Discover. After uv sync --all-packages, Python's entry-point registry knows your parcel.modules entry. The shell's discover_modules() finds it on boot.
  3. Install. An admin approves your declared capabilities via POST /admin/modules/install (or parcel install, or a sandbox promotion). The shell runs your Alembic migrations against the mod_<name> schema and attaches your permissions to the built-in admin role.
  4. Mount. Your router is included under /mod/<name>/, your templates dir is prepended to the Jinja loader, your sidebar items appear in the admin UI.
  5. Upgrade. Ship a new version, bump the Alembic head, run parcel migrate (or the admin endpoint). Your schema migrates; the rest of the shell is undisturbed.
  6. Uninstall. Soft uninstall flips the row inactive (routes stay mounted until the next restart, that's a FastAPI constraint). ?drop_data=true additionally downgrades your migrations and drops your schema.

Schema isolation

Every module owns one Postgres schema. Contacts owns mod_contacts. Your notes module will own mod_notes. The shell owns shell. The public schema is empty by policy. Your Alembic migrations fully qualify the schema in every statement so there's never ambiguity. Cross-schema access is allowed in SQL — nothing stops you joining mod_notes.notes to shell.users if you really want — but you should expose such access through the SDK or a dedicated service function, not a raw SQL join in your router.

Entry points, not magic

There is no plugin registry, no config file listing modules, no filesystem scan. The shell uses Python's standard entry-point mechanism to find modules. Your module's pyproject.toml contains: 

[project.entry-points."parcel.modules"]
notes = "parcel_mod_notes:module"

And that's it. When parcel_mod_notes is importable (via pip install or a uv workspace), the shell sees it. When you pip uninstall it, the shell stops seeing it and flips the installed-module row inactive on next boot.

The one-line rule

If you can only remember one thing from this page, make it this:

Modules import parcel_sdk. Never parcel_shell. The static-analysis gate enforces this. Your runtime code must follow it; your tests are allowed to bypass it (they're not installed into production, they exercise the shell–module boundary from both sides).
