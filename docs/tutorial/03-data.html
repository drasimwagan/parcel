03 Data layer — models and migrations

Now we give the Notes module something to store. One entity — a Note with a UUID primary key, a title, a body, and a created-at timestamp. We'll also write the initial Alembic migration that brings the table into existence.

The model

Open src/parcel_mod_notes/models.py and replace its contents: 

from __future__ import annotations

import uuid
from datetime import datetime

from sqlalchemy import MetaData, Text, func
from sqlalchemy.dialects.postgresql import TIMESTAMP, UUID
from sqlalchemy.orm import DeclarativeBase, Mapped, mapped_column

metadata = MetaData(schema="mod_notes")


class NotesBase(DeclarativeBase):
    metadata = metadata  # type: ignore[assignment]


class Note(NotesBase):
    __tablename__ = "notes"

    id: Mapped[uuid.UUID] = mapped_column(
        UUID(as_uuid=True),
        primary_key=True,
        server_default=func.gen_random_uuid(),
    )
    title: Mapped[str] = mapped_column(Text, nullable=False)
    body: Mapped[str] = mapped_column(Text, nullable=False, default="")
    created_at: Mapped[datetime] = mapped_column(
        TIMESTAMP(timezone=True),
        nullable=False,
        server_default=func.now(),
    )
Conventions you're seeing.
  • from __future__ import annotations at the top. Always.
  • UUID primary keys, generated server-side by Postgres's gen_random_uuid(). No Python-side UUID generation needed.
  • Timestamptz for every datetime column. Naive datetimes will trip you up in production.
  • Full type hints with Mapped[...] — SQLAlchemy 2.0 style, which pyright understands.

The initial migration

The scaffold already created alembic/versions/0001_init.py with just a CREATE SCHEMA. We'll add the notes table to that same revision (since we haven't installed yet, there's no value in splitting into two revisions). Edit it: 

"""init notes

Revision ID: 0001_init
Revises:
"""
from __future__ import annotations

import sqlalchemy as sa
from alembic import op
from sqlalchemy.dialects.postgresql import TIMESTAMP, UUID

revision = "0001_init"
down_revision = None
branch_labels = None
depends_on = None


def upgrade() -> None:
    op.execute('CREATE SCHEMA IF NOT EXISTS "mod_notes"')
    op.create_table(
        "notes",
        sa.Column(
            "id",
            UUID(as_uuid=True),
            primary_key=True,
            server_default=sa.text("gen_random_uuid()"),
        ),
        sa.Column("title", sa.Text(), nullable=False),
        sa.Column("body", sa.Text(), nullable=False, server_default=sa.text("''")),
        sa.Column(
            "created_at",
            TIMESTAMP(timezone=True),
            nullable=False,
            server_default=sa.func.now(),
        ),
        schema="mod_notes",
    )


def downgrade() -> None:
    op.drop_table("notes", schema="mod_notes")
    op.execute('DROP SCHEMA IF EXISTS "mod_notes" CASCADE')

Note the fully-qualified schema="mod_notes" on create_table. Every DDL statement in your migrations should qualify its schema. The sandbox install path depends on this: it mutates your module's metadata.schema in memory to install at mod_sandbox_<uuid>, and ambiguous migrations break.

A small service layer

Create src/parcel_mod_notes/service.py — all the database logic your router needs lives here. Routers stay thin, services do the work. 

from __future__ import annotations

import uuid
from collections.abc import Sequence

from sqlalchemy import select
from sqlalchemy.ext.asyncio import AsyncSession

from parcel_mod_notes.models import Note


async def list_notes(db: AsyncSession) -> Sequence[Note]:
    stmt = select(Note).order_by(Note.created_at.desc())
    return (await db.execute(stmt)).scalars().all()


async def get_note(db: AsyncSession, note_id: uuid.UUID) -> Note | None:
    return await db.get(Note, note_id)


async def create_note(db: AsyncSession, *, title: str, body: str) -> Note:
    note = Note(title=title.strip(), body=body)
    db.add(note)
    await db.flush()
    return note


async def delete_note(db: AsyncSession, *, note: Note) -> None:
    await db.delete(note)
    await db.flush()

Every service function takes a db: AsyncSession and mutates via db.add / db.delete / db.flush. The caller (your router) is responsible for committing — the shell's get_session() FastAPI dep does that automatically at the end of a successful request, and rolls back on exceptions.

Why schema isolation matters

Your tables live in mod_notes. The shell's users, roles, sessions tables live in shell. A different module's tables live in mod_<theirs>. No accidental name collisions — you can have a notes.notes table and another module can have tasks.notes, and they coexist.

You also get a clean uninstall story: parcel uninstall notes --drop-data runs your downgrade, drops the schema, and every row is gone. Nothing to chase through public.

Verify

uv run pytest modules/notes/

The smoke test should still pass — we haven't broken anything. Model code doesn't run until something imports it at request time, so syntax errors in models.py would be silent here. Run the gate to catch them: 

uv run parcel sandbox install ./modules/notes

The sandbox install actually imports your module and runs its migrations. If models.py has a typo, this is where it surfaces. Dismiss the sandbox when you're done (parcel sandbox dismiss <uuid>).
